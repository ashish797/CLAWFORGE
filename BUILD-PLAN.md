# CLAWFORGE — The Definitive Build Plan

## What We're Building

A self-evolving multi-agent system on Telegram. Built on OpenClaw, enhanced with Claude Code's architecture, with a unique 3-layer agent management system.

---

## Part 1: The Architecture

### Three Layers

```
CLAW (Main Agent — Telegram #general)
├── Understands user requirements
├── Finds or creates departments
└── Routes requests

DEPARTMENT (Pod Lead — Telegram #department-name)
├── Forges agents from pool
├── Handles training phase
├── Assigns tasks
└── Reports to CLAW

AGENT (From Pool — works in department topic)
├── Dynamic: re-skinned as needed
├── Production: locked, permanent
├── Trains with user (first time)
└── Executes (after training)
```

### Agent Pool

```
5 runtimes (default):
  [1] 🔒 🎶 Songwriter (PRODUCTION)
  [2] dynamic (idle → re-skinnable)
  [3] 💻 Backend Dev (PRODUCTION)
  [4] dynamic (idle → re-skinnable)
  [5] dynamic (idle → re-skinnable)

Identity = Config swap:
  { name, emoji, department, claw_md, memory_dir, permission_level, type }

Memory persists on disk:
  memory/agents/songwriter/  → songwriter's memories
  memory/agents/tester/      → tester's memories
  Loaded on re-skin, saved on task completion
```

### Training Loop

```
First time: Agent enters TRAINING MODE
  → Shows first draft
  → User tweaks
  → Agent learns preferences
  → Iterates until satisfied
  → Pipeline LOCKED 🔒

After training: Agent EXECUTES directly
  → No back-and-forth
  → Knows user preferences
  → Produces final result
```

## Part 2: Claude Code Enhancements

These enhance agent execution within the 3-layer system:

### 2.1 Conversation Phases
Each agent follows structured phases:
```
PLAN → ARCHITECTURE → GENERATE → TEST → ITERATE
```
Added to system prompt. Agent doesn't generate everything at once.

### 2.2 System Prompt Enhancement
```
System Prompt =
  Project instructions (CLAW.md) +
  Git context (branch, status) +
  Conversation phases instruction +
  Memory index (lightweight pointers) +
  Agent's own CLAW.md (specialized) +
  Existing workspace/skills context
```

### 2.3 Memory Architecture (Per Agent)
```
Layer 1: memory.md (index — lightweight pointers, not full content)
Layer 2: autodream (background consolidation via Chyros cron)
Layer 3: session layer (full conversation history, tool results)
```
Each agent has its own memory directory. Loaded on re-skin.

### 2.4 Permission Modes (5 Levels)
```
ReadOnly → WorkspaceWrite → Prompt → DangerFullAccess → Allow
```
Per-agent permission level. Telegram inline buttons for approvals.

### 2.5 Tool Hooks
```
PreToolUse: runs before tool execution (exit 0=allow, 2=deny)
PostToolUse: runs after tool execution (can modify output)
```
Shell scripts or TypeScript handlers. Configurable per department.

### 2.6 Compaction
```
When context fills up:
  → Summarize old messages
  → Keep last 4 verbatim
  → Inject continuation preamble
  → "Continue directly — do not acknowledge the summary"
```

### 2.7 Chyros (Nightly Agent)
```
Cron at 2 AM:
  → Review all agent memories
  → Extract decisions, preferences, context
  → Update memory files
  → Dream mode (weekly deep consolidation)
  → Send summary to Telegram #general
```

### 2.8 Undercover Mode
```
PostToolUse hook:
  → Strip AI indicators from outputs
  → Clean commit messages
  → No "Generated by AI" comments
```

### 2.9 Buddy (Telegram Reactions)
```
After agent responses:
  → ^_^ "Nice work!" (on success)
  → ?_? "Something went wrong" (on error)
  → -_- "Take a break?" (after 2+ hours)
```

## Part 3: Telegram UX

### Channel Structure
```
#general       → CLAW (entry point, results)
#engineering   → Engineering department + its agents
#multimedia    → Multimedia department + its agents
#research      → Research department + its agents
```
One topic per department. Not one per agent.

### User Interactions
```
Forge flow:
  You: "I want song lyrics"
  Claw: Understands → [Forge 🔨] [Plan 📝]
  You: [Forge🔨]
  Claw: Creates department → Forges agent → Training begins

Permissions:
  Agent: "About to run: rm -rf node_modules"
  Bot: [Approve] [Deny] (inline buttons)

Status:
  /status → Shows active agents, their states, queue

Buddy:
  After agent response: "^_^ Nice work!" (inline message)
```

### What User Sees
```
Clean. Professional. One chat.
Sub-agents work silently.
Results appear as main agent response.
Only surface sub-agent work if: needs approval, fails, or takes too long.
```

## Part 6: Success Criteria

After building, the system should:

1. User talks to CLAW on Telegram
2. CLAW understands requirements
3. CLAW creates department if needed
4. Department forges agent from pool
5. Agent trains with user (first time)
6. Agent executes (after training)
7. Results appear in Telegram
8. Agent returns to pool (or stays if production)
9. Next similar request routes directly to trained agent
10. Chyros improves all agents nightly
